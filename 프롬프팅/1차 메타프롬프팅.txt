메타프롬프팅: 온기한식뷔페 — 회사별 후불 장부 관리 앱 (테블릿용)

아래는 당신(제품기획자/PO)이 LLM(또는 팀원)에게 직접 넣어서 PRD, UI 스펙, API/DB 설계, 관리자 기능, 프라이버시 가이드까지 한 번에 생성하게 할 수 있는 '메타 프롬프트 세트'입니다.
각 블록은 바로 복사해서 LLM에 넣으면 구체적이고 실행 가능한 PRD(및 산출물)를 만들어줍니다.

1) 전체 지시문 (최상위 메타프롬프트 — PRD 생성용)
당신은 경험 많은 제품기획자이자 UX 디자이너이며, 테블릿 기반 업무용 앱의 PRD(제품 요구사항 문서), 화면 스펙, 데이터 모델, 관리자(백오피스) 기능 명세, 최소요건(MVP), 수락기준(acceptance criteria)까지 한 번에 만들어야 한다. 
목표: 점심시간에 20개 내외의 기업 고객이 '후불'로 먹은 인원수들을 간편히 장부에 기록하고, 결제 시 자동으로 합계와 기간을 선택해 '결제 완료' 처리할 수 있는 테블릿 앱.

제약/요구(사용자 제공):
- 디바이스: 테블릿 PC. 화면 가로 기준 레이아웃 권장(또는 세로도 호환).
- 시작 화면 타이틀: "온기한식뷔페 회사별 장부 시스템"
- 로그인: 비밀번호 입력(간단 PIN 방식). 로그인 후 장부 입력 화면 활성화.
- 회사 레코드: 회사명, 회사코드(4자리 숫자, 식당에서 수동 입력), 담당자명, 연락처, 회사비밀번호(옵션), 사업자번호, 회사주소(사업자번호/주소는 선택 저장; 개인정보 절차는 아래 설명).
- 장부 동작:
  - 왼쪽: 드롭다운 / 검색으로 회사 선택 또는 신규 등록.
  - 회사가 선택되고 회사코드와 비밀번호(또는 권한)가 확인되면 오른쪽 장부 영역이 활성화.
  - 인원 드롭다운: 1~20명.
  - 입력 시 자동으로 현재(또는 선택한 연/월)의 날짜에 한 줄씩 추가(날짜 자동/수동 선택 허용).
  - 상단에 연도, 월 드롭다운(기본: 오늘 날짜 기준 2025년 9월).
  - 오른쪽 상단: '결제할 금액', '총 인원 수', '기간(시작일 ~ 종료일)' 표시. 시작일 기본: 마지막 결제 다음날(자동 설정), 종료일 기본: 오늘.
  - 결제 버튼 누르면 확인 팝업(show summary, 예/아니오). 예 선택 시 해당 기간의 항목이 '결제 완료' 상태로 표시(배경색 변경 + 가로 얇은 취소선). 영수증 이미지 붙이기(파일 첨부/사진).
- 비활성 UI: 회사 미선택 시 드롭다운/입력 UI는 회색(비활성)으로 표시.
- 행 레이아웃: 한 달(최대 31일) 표시, 화면 가독성 고려하여 2열 분할(예: 1~16, 17~31) 또는 동적 행 배치.
- 관리자 페이지: 회사 관리(등록/수정/삭제), 결제 내역 조회(회사별/기간별/복수 선택 가능), 총매출 리포트, CSV/엑셀 내보내기, 결제 상태 필터.
- 보안/개인정보: 기본적으로 회사명/담당자/연락처/회사코드/패스워드만 필수. 사업자번호/주소는 선택항목으로 두고 저장 시 암호화 및 로컬 저장 옵션(인터넷 비공개)을 권장. 간단한 가이드 추가.

생성할 산출물:
1. PRD(목표, 배경, 사용자 스토리, MVP, 비기능 요구사항)
2. 화면 흐름도(화면 목록 + each 화면의 구성요소, 활성/비활성 상태 설명)
3. 상세 UI 스펙(테블릿 가로/세로 — 각 컴포넌트의 레이블, 기본값, 동작)
4. 데이터 모델(테이블/스키마: Company, Entry, Payment, User(관리자))
5. 간단한 API 명세(엔드포인트, 입력/출력 샘플)
6. 수락기준(테스트 가능한 항목)
7. 개인정보 저장 가이드(권장, 최소, 암호화 수준)
8. 배포/테스트 체크리스트(테블릿에서의 테스트 포인트)

출력 형식:
- PRD는 목차 포함 Markdown.
- 화면 스펙은 각 화면별 섹션(헤더 H1)로 정리.
- 데이터 모델은 표나 JSON 스키마.
- 수락기준은 항목화된 체크리스트.

지금부터 위 요구사항을 반영하여 PRD와 관련 산출물을 생성하라. 한국어로 작성하라.

2) UI / 화면 설계 생성용 프롬프트 (Figma-ready)
당신은 UI 디자이너다. 아래 요구사항으로 **테블릿(가로 우선)** 디자인 명세(컴포넌트 단위로 Figma에 바로 옮길 수 있는 수준)를 만들어라.

요구사항 요약:
- 해더: 왼쪽에 연도/월 드롭다운(기본: 오늘 → 2025 / 9), 중앙에 타이틀 "온기한식뷔페 회사별 장부 시스템", 오른쪽에 요약(총 인원, 결제금액, 기간 선택 드롭다운, 결제 버튼).
- 레이아웃: 좌측 패널(폭 30~35%) — 회사 선택/신규 등록/회사코드 입력/인원 드롭다운(1~20). 우측 패널(폭 65~70%) — 장부 테이블(날짜/수량/서명/영수증 첨부 버튼), 기본 설명 텍스트(비활성 시).
- 비활성 상태: 좌측에서 회사가 미선택이면 우측은 회색 오버레이, 설명 텍스트로 사용법 표시.
- 입력 동작:
  - 회사선택 → 회사코드 자동포커스 → 코드 입력 후 '인증' 버튼(또는 내부 매칭) → 성공 시 우측 활성화.
  - 인원 선택 + '등록' 버튼 → 우측 장부에 (선택한 연/월의 선택된 날짜 라인에) 자동으로 행이 추가(또는 오늘 날짜로 추가). 날짜명 표기는 "25.09.28".
  - 장부에서 멀티 선택(체크박스)로 기간 선택 후 '결제하기' → 확인 팝업(요약, 영수증 첨부 옵션, 예/아니오).
  - 결제 완료 시 해당 행들에 연한 회색 배경 + 얇은 가로 취소선 표시 + 영수증 아이콘(클릭 시 보기).
- 테이블 레이아웃: 한 달을 두 컬럼으로 나눔(좌: 1~16 / 우: 17~31) 또는 동적 그리드. 각 셀은 날짜, 총 인원, 마지막 서명(텍스트), 영수증 스택.
- 관리자 페이지: 회사 목록(검색, 필터), 회사 등록 폼, 결제 내역 대시보드(필터: 회사, 기간, 결제상태), CSV export 버튼, 총매출 카드.
- 색상/타이포: 클린한 카드 UI, 가독성 높은 폰트, 버튼 컨트라스트 강조, 비활성은 #E0E0E0 느낌.

출력: 각 화면별로 컴포넌트 이름, 위치(좌/우/상단), 기본 텍스트, 기본값, 상호작용(예: 클릭 시, 비활성 조건), 그리고 Figma에서 바로 만들 수 있는 간단 와이어프레임 텍스트 설명을 제공하라. 한국어.

3) 데이터 모델 / API 스펙 생성용 프롬프트
당신은 백엔드 개발자다. 위 앱을 위해 간단하고 안전한 데이터 모델 및 REST API 스펙을 작성하라. 로컬 또는 사내 배포용을 상정하라.

DB 테이블(권장): Company, Entry, Payment, AdminUser
각 테이블에 대한 필드, 타입, 필수여부, 인덱스 제안, 샘플 SQL 생성문을 포함하라.

예시 API:
- POST /login (body: {pin}) -> 토큰
- GET /companies -> 회사 리스트
- POST /companies -> 회사 등록
- POST /companies/{id}/entries -> 한 줄 추가 (date, count, signer)
- GET /companies/{id}/entries?year=2025&month=9 -> 월별 데이터
- POST /companies/{id}/payments -> 결제 처리 (entry_ids[], receiptImage)
- GET /reports/sales?from=2025-09-01&to=2025-09-28 -> 합계
각 엔드포인트의 요청/응답 예시(JSON), 에러 코드(간단) 포함.

4) 관리자/운영 요구 프롬프트 (리포트 및 CSV)
관리자 관점에서의 기능을 상세히 만들어라:
- 회사 등록/수정/삭제 플로우(필드: 회사명, 담당자명, 연락처, 회사비밀번호, 사업자번호(옵션), 주소(옵션))
- 결제 내역 조회 UI 및 동작(복수 선택, 기간 필터, 전체선택)
- 리포트: 총매출(기간별), 회사별 매출, 결제 미완료 내역
- CSV/엑셀 export 형식(컬럼명 / 샘플 row)
- 감사 로그: 누가 언제 결제 처리했는지 기록
- 권한: 관리자(전체 권한) / 카운터(입력·결제 권한만) — 권한별 가능한 액션 표기

5) 개인정보 & 보안 가이드 프롬프트
법률 자문은 아니지만 실무 권장안을 제공하라. 
- 필수 저장 정보(회사명, 회사코드, 담당자 이름, 연락처, 내부 패스워드/핀)
- 선택 저장 항목(사업자번호, 주소) — 보관 시 위험 및 권장 암호화 방식(AES-256 권장, 전송 시 TLS), 로컬 저장(인터넷 미연결) 옵션 권장
- 암호화/해싱: 관리자/회사 비밀번호는 bcrypt(또는 Argon2)로 해싱, 사업자번호 등 민감 데이터는 암호화 저장
- 백업, 접근 제어, 로그 보관 기간(예: 3년/1년 등 권장), 데이터 삭제 정책(사용자가 삭제 요청 시)
- 최소 권장: 내부망 또는 사내 서버에 호스팅, 외부 클라우드 사용 시 접근 제한 및 암호화

6) Acceptance Criteria (테스트 가능한 항목) 프롬프트
MVP에 대한 수락 기준을 작성하라(자동화/수동 테스트 케이스 포함).
예:
- 로그인: 올바른 PIN 입력 시 세션 발급, 잘못된 PIN 3회 이상 시 5분 잠금
- 회사 등록: 필수 필드 입력 시 저장 확인
- 장부 입력: 회사 선택 → 인원 선택 → '등록' 클릭 시 해당 날짜에 항목 추가
- 결제 프로세스: 기간 설정 → 결제 버튼 → 확인 팝업 → '예' 선택 시 해당 항목들 상태가 결제 완료로 변경(배경색+취소선) 및 결제 내역에 레코드가 생성
- 리포트: 기간별 매출 합계가 올바르게 계산되는지(예: 3+4+8에 8000원 곱셈 검증)
- 파일 첨부: 영수증 첨부 후 보기 가능
- 관리자: 회사별 결제 내역 필터/CSV 추출 가능

7) 산출물 요청 예시 (한 번에 여러 요청)
아래 3가지를 한 번에 출력해줘:
1) PRD (Markdown, 목차 포함)
2) 테블릿 메인 화면과 관리자 화면의 상세 UI 스펙(컴포넌트 단위)
3) DB 스키마 + API 명세(요청/응답 예시)
출력은 한국어로, 명확한 섹션 헤더(H1)를 사용해서 구분해라.

8) 개인정보 수집 범위에 대한 간단 권고 (출력에 포함)

권장 수집 최소값(필수): 회사명, 회사코드(4자리), 담당자명(옵션), 연락처(옵션), 내부 PIN/비밀번호(저장: 해싱)

옵션(사업자 필요시만): 사업자번호, 회사주소 — 저장 시 암호화, 법적 보관 이유(세금/회계) 없으면 수집하지 말 것

보관 주기 권장: 결제 관련 자료는 회계 규정에 따라 보관(대한민국 일반: 5년 권장), 그러나 개인정보는 최소 보관

권장 기술: TLS for transport; bcrypt/argon2 for passwords; AES-256 for sensitive fields; DB 접근 권한 최소화

마무리 안내 (사용자에게 보여줄 멘트)
